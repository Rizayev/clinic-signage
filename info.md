# Анализ идеи и ТЗ

## Система управления контентом для ТВ/мониторов в клинике

## 1. Суть идеи

Нужно сделать централизованную систему, через которую администратор клиники сможет управлять всеми экранами в филиале или сети филиалов.

На экранах могут показываться:

* рекламные видео;
* акции и скидки;
* бегущая строка;
* баннеры;
* расписание врачей;
* информационные сообщения;
* время, дата, погода;
* правила клиники;
* уведомления для пациентов;
* брендированное оформление;
* контент по зонам: ресепшен, коридор, кабинет, ожидание, лаборатория и т.д.

Главная идея:
**в панели администратор видит схему клиники, понимает где какой ТВ находится, назначает контент на конкретный экран или группу экранов, задаёт расписание показа и отслеживает статус устройств.**

---

# 2. Ключевые сценарии использования

## Сценарий 1: Управление экранами по схеме клиники

Администратор открывает панель и видит план клиники:

```text
Ресепшен
 ├── TV-01
 └── TV-02

Зал ожидания
 ├── TV-03
 ├── TV-04
 └── TV-05

Коридор 2 этаж
 ├── TV-06
 └── TV-07
```

Или визуально:

```text
[Вход]     [TV-01]
[Ресепшен] [TV-02]

[Зал ожидания]
[TV-03] [TV-04] [TV-05]

[Кабинет УЗИ] [TV-06]
```

Администратор нажимает на ТВ и видит:

* название устройства;
* где находится;
* онлайн/оффлайн;
* что сейчас показывает;
* текущий плейлист;
* яркость/громкость, если поддерживается;
* последнее подключение;
* IP/MAC/Android ID;
* версия приложения;
* ошибки воспроизведения.

---

## Сценарий 2: Загрузка видео и баннеров

Администратор загружает:

* MP4-видео;
* изображения JPG/PNG/WebP;
* HTML-баннеры;
* текстовые сообщения;
* готовые шаблоны акций;
* логотип клиники;
* фоновые изображения.

После загрузки файл попадает в медиатеку.

В медиатеке можно:

* переименовать файл;
* посмотреть превью;
* назначить категорию;
* указать длительность показа;
* активировать/деактивировать;
* удалить;
* заменить файл без перенастройки плейлиста.

---

## Сценарий 3: Создание плейлиста

Администратор создаёт плейлист:

**Плейлист: “Зал ожидания — утро”**

| Порядок | Контент               | Тип       | Длительность |
| ------: | --------------------- | --------- | -----------: |
|       1 | Видео про имплантацию | Видео     |       30 сек |
|       2 | Скидка 20% на анализы | Баннер    |       15 сек |
|       3 | График врачей         | Инфо-блок |       20 сек |
|       4 | Бегущая строка        | Overlay   |    постоянно |
|       5 | Видео про check-up    | Видео     |       45 сек |

Плейлист можно назначить:

* на один ТВ;
* на группу ТВ;
* на зону;
* на филиал;
* на все устройства.

---

## Сценарий 4: Расписание показа

Можно задать:

* показывать каждый день;
* показывать только по будням;
* показывать в конкретные часы;
* показывать с даты по дату;
* показывать каждые N минут;
* показывать после каждого X ролика;
* показывать только утром/днём/вечером;
* показывать срочное сообщение поверх всего.

Пример:

```text
Акция “Скидка на стоматологию”
Период: 01.06.2026 — 30.06.2026
Время: 09:00 — 18:00
Частота: каждые 10 минут
Экраны: TV-01, TV-02, TV-03
```

---

## Сценарий 5: Бегущая строка

Администратор вводит текст:

```text
Уважаемые пациенты! Приём ведётся по предварительной записи. Скидка 15% на check-up до конца месяца.
```

Настройки:

* скорость движения;
* цвет текста;
* цвет фона;
* прозрачность;
* позиция: сверху/снизу;
* показывать всегда или по расписанию;
* показывать только на выбранных ТВ;
* несколько сообщений по очереди.

---

## Сценарий 6: Экстренное сообщение

Например:

```text
Внимание! Сегодня лаборатория работает до 16:00.
```

Администратор нажимает “Показать срочно”, и сообщение появляется поверх всего контента на выбранных экранах.

## 4.4. Схема клиники

Схема может быть реализована в двух вариантах.

### Вариант 1: Простая схема

Сначала можно сделать без сложного редактора плана помещения.

Например:

```text
Филиал → Зоны → ТВ
```

Это быстро и достаточно для MVP.

---

### Вариант 2: Визуальная схема

Администратор загружает картинку/план клиники и расставляет ТВ как точки.

Функции:

* загрузка изображения плана;
* добавление маркеров ТВ;
* drag & drop;
* подпись ТВ;
* цвет статуса:

  * зелёный — онлайн;
  * красный — оффлайн;
  * жёлтый — ошибка;
* клик по ТВ открывает карточку устройства.

Это хороший вариант для полноценной версии.

# 7. Требования к Android TV приложению

Для Android TV/Android Box лучше сделать отдельное player-приложение.

## Приложение должно уметь:

* регистрироваться в системе по коду;
* получать настройки с сервера;
* скачивать медиафайлы локально;
* кэшировать контент;
* проигрывать видео;
* показывать изображения;
* показывать HTML/баннеры;
* показывать бегущую строку поверх видео;
* работать оффлайн по последнему загруженному плейлисту;
* отправлять heartbeat на сервер;
* отправлять ошибки;
* автоматически запускаться после перезагрузки ТВ/приставки;
* скрывать Android интерфейс;
* обновляться через APK или MDM;
* показывать код привязки при первом запуске.

---

## Регистрация устройства

При первом запуске приложение показывает код:

```text
Код устройства: A7K9-22
```

Администратор в панели нажимает:

```text
Добавить устройство → Ввести код A7K9-22 → Назвать TV-01 → Выбрать зону
```

После этого устройство привязано к клинике.

---

## Heartbeat

Каждые 10–30 секунд приложение отправляет:

```json
{
  "device_id": 15,
  "status": "online",
  "current_media_id": 82,
  "playlist_id": 7,
  "app_version": "1.0.3",
  "free_storage": 12400000000,
  "ip_address": "192.168.1.45"
}
```

Если heartbeat не приходит больше, например, 2 минут — устройство считается offline.

# 8. Важный момент по Chromecast

Chromecast — не такой удобный вариант, как Android TV/Android Box.

## Проблема Chromecast

Chromecast обычно работает как приёмник трансляции. Он не всегда удобен для автономной работы 24/7, потому что:

* ему нужен отправитель контента;
* сложнее сделать локальный кэш;
* сложнее управлять расписанием;
* сложнее мониторить состояние;
* меньше контроля над устройством;
* сложнее автозапуск после сбоя;
* сложнее обновлять player.

## Рекомендация

Для нормальной системы лучше использовать:

```text
Android TV
или
Android TV Box
или
мини-ПК/Raspberry Pi
```

Chromecast можно поддержать как дополнительный вариант, но не как основной.

Лучший вариант для клиники:

```text
Android TV Box + установленное player-приложение
```

Преимущества:

* дешевле;
* можно поставить на любой монитор;
* можно кэшировать видео;
* можно запускать приложение при старте;
* можно управлять устройством;
* можно смотреть статус онлайн/оффлайн.


* удобно для CRUD;
* можно быстро сделать медиатеку, устройства, плейлисты.

# 14. Логика приоритетов показа

Очень важно заранее определить, что показывается, если на ТВ назначено сразу несколько правил.

Рекомендуемый порядок:

```text
1. Срочное сообщение
2. Контент, назначенный напрямую на устройство
3. Контент, назначенный на зону
4. Контент, назначенный на филиал
5. Дефолтный плейлист
```

Пример:

Если на весь филиал назначен общий плейлист, но на TV-01 назначен отдельный плейлист, то TV-01 показывает свой отдельный плейлист.

# 15. Оффлайн-режим

Это критично.

Если интернет пропал, ТВ не должен показывать чёрный экран.

Android Player должен:

* хранить последний актуальный плейлист;
* хранить скачанные видео/изображения;
* продолжать воспроизведение локально;
* периодически пытаться подключиться к серверу;
* после восстановления связи получить новую конфигурацию.

Если файл ещё не скачался, player должен его пропустить и показать следующий доступный файл.

---

# 16. Требования к стабильности

Система должна быть рассчитана на работу 24/7.

Минимальные требования:

* player автоматически запускается после перезагрузки;
* player не падает при отсутствии интернета;
* если видео повреждено — пропускает его;
* если плейлист пустой — показывает fallback-экран;
* если сервер недоступен — работает по кэшу;
* heartbeat каждые 10–30 секунд;
* логирование ошибок;
* локальное ограничение кэша, например 5–20 GB;
* автоматическая очистка старых файлов.

# 22. Риски

## Риск 1: Chromecast

Если делать упор на Chromecast, проект может стать нестабильным. Лучше использовать Android TV Box.

## Риск 2: Нестабильный интернет

Нужен локальный кэш на устройстве.

## Риск 3: Большие видео

Нужно ограничить формат и размер видео. Желательно конвертировать всё в H.264 MP4.

## Риск 4: Разные разрешения экранов

Нужно поддержать:

```text
1920x1080
3840x2160
vertical 1080x1920
```

## Риск 5: Человеческий фактор

Администратор может назначить неправильный плейлист не туда. Нужны предпросмотр и понятная схема.

## Риск 6: Чёрный экран

Нужен fallback-контент:

```text
Логотип клиники
Информационный экран
Последний рабочий плейлист
```

# 24. Итоговая формулировка проекта

Разрабатывается система Digital Signage для клиники, состоящая из web-панели управления и Android TV player-приложения. Система позволяет централизованно управлять всеми мониторами клиники, видеть их расположение по зонам, загружать рекламный и информационный контент, создавать плейлисты, задавать расписание показа, показывать бегущие строки, баннеры и срочные сообщения, а также отслеживать статус устройств в реальном времени.

Главный фокус системы — стабильное воспроизведение контента на ТВ 24/7, удобное управление для администратора и возможность масштабирования на несколько филиалов.
